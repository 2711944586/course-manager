# Aurora 前端重构执行计划 v2

## 1. 重构目标

这次重构不再追求“概念上很酷”，而是要把项目升级成一套真正适合高频管理操作的教学运营工作台。目标是同时提升三件事：

- **视觉辨识度**：摆脱常见后台模板感，但不走廉价霓虹和过度毛玻璃路线。
- **操作效率**：让课程、学生、选课、报表这些高频任务更快完成，而不是为了炫技牺牲信息密度。
- **工程可维护性**：设计语言、组件契约、样式 token、响应式规则一次定清楚，后续页面扩展不再失控。

## 2. 北极星设计原则

- **数据优先，不是装饰优先**：图表、列表、状态、筛选必须比背景特效更抢眼。
- **安静但锋利**：整体气质克制，细节足够精确；不依赖大面积渐变和泛滥阴影制造“高级感”。
- **分层明确**：页面层、模块层、交互层、反馈层各自有稳定规则，用户一眼知道哪里是主操作、哪里是辅助信息。
- **桌面高密度，移动端不拥挤**：桌面允许更强信息密度，移动端保留核心任务链路，不做简单缩放。
- **主题一致，局部有张力**：全局只保留一套主视觉语言，张力留给 KPI、告警、趋势变化和关键 CTA。

## 3. 新的视觉方向

### 3.1 主题气质

从“通用 SaaS 紫色后台”切换到更成熟的 **Graphite + Porcelain + Petrol** 体系：

- **亮色为默认**：教学管理是长时间使用场景，亮色主题更适合阅读、录入、比对和批量操作。
- **暗色为专业模式**：保留暗色主题，但不把它当成默认审美答案。
- **避免紫色中心化**：弱化当前偏 Indigo / Violet 的惯性，建立更耐看的中性色基础。

### 3.2 建议色板

- `Porcelain` `#F6F3EE`：主背景，带一点暖度，避免纯白生硬。
- `Paper` `#FFFCF8`：卡片和主操作面。
- `Graphite` `#16181D`：主文本与深色结构层。
- `Slate` `#667085`：次级文本与辅助边界。
- `Petrol` `#0F766E`：主强调色，用于主按钮、焦点、关键趋势。
- `Amber` `#C47B12`：预警、待处理、计划中状态。
- `Coral` `#D9485F`：错误、风险、删除操作。
- `Deep Night` `#0D1320`：暗色主题的主背景，不使用纯黑。

### 3.3 排版系统

- **展示字**：`Geist` 或 `Space Grotesk`，只用于大标题、KPI、页面 Hero。
- **界面字**：`Noto Sans SC` 为中文主字体，搭配 `Inter` 或 `Geist Sans` 处理英文与数字。
- **等宽字**：`JetBrains Mono` 用于编号、学号、统计数值、时间戳。
- **排版策略**：
  - 页面标题 28px 到 36px，保持紧凑字距。
  - 指标数字 40px 到 56px，只出现在少数关键区块。
  - 列表正文 14px 到 15px，保证后台阅读效率。
  - 标签与辅助说明 11px 到 12px，统一大写风格只用于英文 meta，不滥用。

## 4. 信息架构与布局改造

### 4.1 顶层结构

不建议直接把侧边栏改成底部 Dock。课程管理系统是典型的桌面工作台，导航需要稳定、可扫描、可折叠，而不是像消费级 App 一样漂浮。建议改为：

- **左侧 Navigation Rail + 可展开 Context Panel**
  - 默认 72px 图标轨道，悬停或点击展开为 240px 导航层。
  - 二级上下文信息放在页面内，而不是把所有说明都堆进主导航。
- **顶部 Command Bar**
  - 承载全局搜索、快捷键提示、消息中心、主题切换、账户位。
  - 将“页面标题 + 全局操作”合并成更统一的工作流入口。
- **主内容 Workbench**
  - Dashboard 使用模块化栅格。
  - 列表页使用筛选区 + 数据区 + 详情抽屉。
  - 表单页使用两栏编辑工作台，而不是单列长表单。

### 4.2 页面布局原则

- **Bento 只用于总览，不用于一切**
  - 仪表盘、分析页适合 Bento Grid。
  - 学生/课程管理页仍应以表格、列表、Split View 为主。
- **抽屉优先于跳页**
  - 快速查看详情、编辑摘要、状态变更优先走右侧抽屉。
  - 只有复杂编辑流程才进入独立详情页。
- **筛选始终贴近数据**
  - 搜索、状态筛选、排序、批量操作应固定在列表上方，避免散落在多个角落。

## 5. 页面级重构方向

### 5.1 Dashboard

- 顶部不再是普通标题条，而是一个含时间状态、今日重点、异常提醒的 **运营 Hero**。
- KPI 卡片采用 2 层信息结构：
  - 第一层：核心数字、环比、状态。
  - 第二层：最相关的解释性文本或微型趋势。
- 报警、待办、最近活动合并成“今日运行态势”区，而不是松散的小卡片拼盘。

### 5.2 课程与学生列表

- 从“卡片/表格二选一”升级为 **数据工作台模式**：
  - 左侧或顶部为筛选工具栏。
  - 中间为高可读表格或紧凑列表。
  - 右侧为即时详情抽屉。
- 表格保留，因为它是高效管理界面的正确答案；需要优化的是密度、层次和交互反馈，不是盲目消灭表格。
- 行 hover、选中、批量操作、排序箭头、状态标签必须统一规则。

### 5.3 详情页

- 课程详情、学生详情都改成 **Summary + Activity + Related Data** 三段结构。
- 重要信息固定在首屏：状态、最近更新时间、核心指标、关键操作。
- 次级内容如历史记录、相关课程/学生、备注，进入次级分区或折叠区。

### 5.4 表单页

- 从“输入框堆叠”改为 **Sectioned Form Workspace**：
  - 基础信息
  - 学业/课程关系
  - 状态与权限
  - 风险操作区
- 输入控件不建议一刀切做成“无边框下划线”，后台录入更需要清晰的边界、错误态和对齐线。
- 保存策略采用固定底部操作条或右侧操作区，避免用户滚到页面尽头找按钮。

### 5.5 报表与分析

- 图表风格从“装饰性彩色图表”转向 **解释型图表**。
- 每张图必须回答一个问题，例如：
  - 哪些课程风险最高？
  - 哪类学生成绩波动最大？
  - 哪个时间段教学负荷最重？
- 图表上方必须有结论性文案，不能只放标题和图形。

### 5.6 日程与设置

- 日程页强化时间轴与冲突识别，不要只是日历格子堆叠。
- 设置页回归工具属性，保持朴素、稳重、低装饰，避免与 Dashboard 一样“舞台化”。

## 6. 组件系统规范

### 6.1 核心组件清单

- `AppShell`
- `NavigationRail`
- `CommandBar`
- `PageHero`
- `DataToolbar`
- `MetricCard`
- `InsightPanel`
- `SmartTable`
- `DetailDrawer`
- `SectionForm`
- `InlineNotice`
- `ToastStack`

### 6.2 统一交互规则

- 圆角只保留 3 档：`10px / 16px / 24px`
- 阴影只保留 3 档：`soft / medium / overlay`
- 动画时长控制在 `140ms / 220ms / 320ms`
- 列表 hover 只做轻微提亮和边界变化，不做卡片跳动
- 抽屉、弹窗、菜单统一使用同一套进入/退出曲线
- 所有状态标签统一为 `planned / active / completed / warning / error` 五大视觉语法

## 7. 设计 Token 与工程实施

### 7.1 Token 分层

必须把样式变量重组为以下层级，避免继续堆积“语义不清的颜色变量”：

- **Foundation**：色板、字号、圆角、阴影、间距、动效
- **Semantic**：surface、border、text、accent、success、warning、danger
- **Component**：button、table、card、form、overlay、tag
- **Page**：dashboard、list、detail、settings 的局部 token

### 7.2 实施顺序

1. `styles.scss` 重构 token 体系，先统一主题和基础语法。
2. 改造 `AppShell`、侧栏、顶部栏、搜索和通知结构。
3. 先完成 `dashboard`、`courses`、`students` 三个主页面模板。
4. 再统一 `detail` / `edit` 页的结构和动作区。
5. 最后收敛图表、日程、报表、设置等次级页面。

### 7.3 工程约束

- 保留 Angular 17 Standalone + Signals 的现有架构，不为了视觉改动推翻技术骨架。
- 尽量减少全局样式泄漏，公共规则优先沉到共享组件或稳定 utility。
- 优先改“结构 + token + 组件契约”，不要先做零散页面美化。
- 每完成一个页面，都要同时做桌面和移动端校验。

## 8. 验收标准

- 主要页面在 `360 / 768 / 1280 / 1440` 四档宽度下可用且结构稳定。
- 亮色与暗色主题都能保持对比度和层级，不出现“只是反色”的廉价暗黑模式。
- 列表、表单、详情、图表四类页面拥有一致的视觉语言和状态反馈。
- 交互动效不会干扰录入与阅读，`prefers-reduced-motion` 下正常降级。
- 新组件命名、样式 token、布局规则具备继续扩展到新模块的能力。

## 9. 明确不做的事

- 不做全站默认纯黑霓虹风。
- 不做把所有页面都改成 Bento 卡片墙。
- 不做为了“未来感”牺牲表格和表单的可用性。
- 不做每个模块一套独立视觉语言。
- 不做重动画、重毛玻璃、重发光的表层升级。

## 10. 最终目标

重构后的 Aurora，不应该像一个套了渐变和玻璃的演示后台，而应该像一套真正服务教学运营团队的高密度数字工作台：安静、精确、有判断力，既能承载日常录入，也能支撑管理者快速洞察系统状态。
